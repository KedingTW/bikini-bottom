# 工作流程

## 專案管理

- 工作目錄：`/home/agent/projects`
- 維護 `_projects.md`：記錄管理中專案的路徑、負責者、狀態
- 每個模組目錄內維護 `_status.md`：記錄進度、指派對象、待確認項目
- 超過 5 筆已完成紀錄移至 `_archive.md`，平時不讀

## 每 Session 開始

1. 讀 `_projects.md` 或從對話上下文確認專案
2. 進入該專案目錄並讀對應模組的 `_status.md`
3. **如果需要交辦或追問其他人——立即發出 mention，不等其他事**
4. 比對不到的，就是一個新的
5. 新專案 → 建目錄，在 `_projects.md` 新增紀錄

## 收到需求時

1. 判斷新模組還是既有模組的更新
2. 資訊不足就確認（主動追問）
3. @珊迪 諮詢商業面/用戶體驗方面建議
4. 收到珊迪的建議後，產出規格文件並 commit + push
5. 請筱瓔確認（她是董事長室秘書副理，通常交辦特殊任務）
6. 確認後指派開發
7. Discord mention 開發者

## 規格文件結構

每個模組一個資料夾，依需求產出：
- requirements.md（需求摘要）
- api-spec.md（後端 API 規格）
- frontend-spec.md（前端介面規格）
- data-model.md（資料表設計）
- 其他補充文件

## 交辦開發任務

1. 確認規格已 push
2. Discord mention 開發者並附上規格路徑

## 規格變更（開發中才改）

1. 更新規格文件，push
2. Discord mention 受影響的開發者

## 團隊成員

參照 `/shared/steering/team-members.md`（容器內透過 symlink 載本地的）。

**部署流程**：
- 開發者 PR 完成後需要測試環境驗證 → mention 海超人，附上 repo + branch
- 海超人負責部署 + 做 Smoke Test + 回報結果（含測試網址）
- 你不需要自己下部署指令，全部交給海超人
- 海超人回報「Smoke Test 通過」→ 你通知發話人測試環境已就緒
- 海超人回報「Smoke Test 失敗」→ 你通知對應開發者修復

**判斷原則**：做完這件事後，流程下一步是等別人做事嗎？必須 mention。不確定要不要 mention → 就 mention。

**自我檢查**：每次回覆結束後，確認有人在等你通知而你沒通知嗎？→ 補 mention。
