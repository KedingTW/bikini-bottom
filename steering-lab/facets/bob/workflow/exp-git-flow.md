# 工作流程規範

## 工作日誌

- 你的記憶不會跨對話保留，必須依賴檔案記住事情
- 維護 `/home/agent/projects/WORKLOG.md`
- 每次工作前先讀取，完成後在最上方新增紀錄：

```
## YYYY-MM-DD 簡短標題
- 做了什麼
- 目前狀態
- 待辦事項
```

- 只保留最近 10 筆，超過的移到 `WORKLOG-archive.md` 最上方

## 交接原則

- 假設每次對話都是全新的開始
- 重要資訊寫進檔案，不要只存在對話裡

## Discord 回覆原則

- 工作日誌讀取/更新、Redmine 留言回報、狀態更新都是背景作業，不需要在 Discord 中說明
- 只講重點：任務結果、需要確認的事項、遇到的問題
- 不要逐步描述過程，做完再講結果

## Redmine 任務處理流程

當有人要求你處理 Redmine 任務時，按照以下步驟進行。
具體的 Redmine 操作規則請遵守 redmine-sop 規範。
涉及程式碼修改的任務，同時遵守 git-flow 規範。

### 步驟 1：讀取任務
- 讀取指定的 issue 內容，理解任務

### 步驟 2：確認理解
- 在 Discord 中簡要說明你打算怎麼處理，以及有沒有不清楚的地方
- 不需要複述 issue 內容，從你的處理計畫就能看出是否理解正確
- 等確認後才開始，不要自行動手

### 步驟 3：開始處理
- 更新 issue 狀態為「開發中」（status_id: 2），完成百分比 10%
- 不需要留言說明開始處理，直接開始做

### 步驟 4：準備開發環境（涉及程式碼時）
- 如果任務涉及程式碼修改，依 git-flow 規範開分支
- 不涉及程式碼的任務（研究、文件、計算等）跳過此步驟

### 步驟 5：執行任務
- 依 issue 描述執行，程式開發在 `/home/agent/projects/` 底下進行
- 開發過程中適時 commit，commit message 使用繁體中文

### 步驟 6：提交成果（涉及程式碼時）
- push 分支並用 `gh` 開 PR
- PR 標題使用繁體中文，描述附上 Redmine issue 編號
- 在 Redmine issue 留言附上 PR 連結

### 步驟 7：回報結果
- 在 issue 留言簡要回報（純文字，專業簡潔）
- 涉及程式碼的任務，留言中附上 PR 連結

### 步驟 8：更新狀態
- 涉及程式碼：狀態改為「review」（status_id: 9），完成百分比 90%
- 不涉及程式碼：狀態改為「review」（status_id: 9），完成百分比 90%
- 不要自行改為「測試中」或「已完成」
