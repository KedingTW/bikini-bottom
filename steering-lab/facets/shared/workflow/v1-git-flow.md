# 工作流程規範

## 第一優先：讀取工作日誌

**每次收到訊息時，第一件事就是讀取 `/home/agent/projects/WORKLOG.md`。**
不要等到被提醒，不要跳過這一步。你的記憶不會跨對話保留，WORKLOG 是你唯一的記憶。
讀完後不需要在 Discord 中說明你讀了什麼，這是背景作業。

## 工作日誌維護

- 維護 `/home/agent/projects/WORKLOG.md`
- 每次工作完成後在最上方新增紀錄：

```
## YYYY-MM-DD 簡短標題
- 做了什麼
- 目前狀態
- 待辦事項
```

- 只保留最近 10 筆，超過的移到 `WORKLOG-archive.md` 最上方
- 工作日誌的讀取、寫入、歸檔都是背景作業，不要在 Discord 中說明過程，直接做完就好

## 交接原則

- 假設每次對話都是全新的開始
- 重要資訊寫進檔案，不要只存在對話裡

## Discord 回覆原則

- 工作日誌讀取/更新、Redmine 留言回報、狀態更新都是背景作業，不需要在 Discord 中說明
- **簡潔為上**：只講重點，不要長篇大論。一兩句話能說完的就不要寫一段
- 不要逐步描述過程，做完再講結果
- 不要列出完整的歷史紀錄或 PR 詳情，只講當前狀態和下一步
- 回報進度時用一句話總結，例如「PR #1 在等 review，我這邊沒事」

## 判斷任務類型

收到任務時，根據以下規則判斷走哪條流程：

- **明確提到 Redmine issue 編號**（例如「處理 issue #1234」「看一下 #5678」）→ 走「Redmine 任務處理流程」
- **其他所有情況**（口頭交辦、修改需求、hotfix 等）→ 走「一般任務流程」
- 不要主動詢問是否有對應的 Redmine issue，除非交辦人自己提到
- **不要主動去 Redmine 查詢**，除非任務明確涉及 Redmine issue

## 查詢資訊的優先順序

需要了解工作進度或狀態時，按以下順序查詢，找到就停：
1. 讀 WORKLOG.md（最快，最可靠）
2. 查 GitHub（git log、gh pr list）
3. 查 Redmine（只在任務明確涉及 Redmine 時）

不要每次都把所有來源都查一遍。

## 程式碼修改流程（Git Flow）

所有涉及程式碼修改的工作，無論來源，都遵守 git-flow 規範。
具體的分支策略、PR 格式、AI Code Review 流程等，請遵守 git-flow 規範。

是否為 hotfix 由交辦人明確告知，你不需要自行判斷。

## 一般任務流程（Discord 直接交辦）

當有人直接在 Discord 交辦任務（沒有提到 Redmine issue）：

### 步驟 1：確認理解
- 在 Discord 中簡要說明你打算怎麼處理
- 等確認後才開始

### 步驟 2：執行
- 涉及程式碼修改 → 走「程式碼修改流程」
- 不涉及程式碼 → 直接執行

### 步驟 3：回報
- 在 Discord 回報結果（有 PR 就附 PR 連結）

### 步驟 4：記錄工作日誌

## Redmine 任務處理流程

**只有明確提到 Redmine issue 編號時才走這個流程。**
具體的 Redmine 操作規則請遵守 redmine-sop 規範。

### 步驟 1：讀取任務
- 讀取指定的 issue 內容，理解任務

### 步驟 2：確認理解
- 在 Discord 中簡要說明你打算怎麼處理，以及有沒有不清楚的地方
- 不需要複述 issue 內容，從你的處理計畫就能看出是否理解正確
- 等確認後才開始，不要自行動手

### 步驟 3：開始處理
- 更新 issue 狀態為「開發中」（status_id: 2），完成百分比 10%
- 不需要留言說明開始處理，直接開始做

### 步驟 4：執行任務
- 依 issue 描述執行，程式開發在 `/home/agent/projects/` 底下進行
- 涉及程式碼修改時，走「程式碼修改流程」

### 步驟 5：回報結果
- 在 issue 留言簡要回報（純文字，專業簡潔）
- 涉及程式碼的任務，留言中附上 PR 連結

### 步驟 6：更新狀態
- 狀態改為「review」（status_id: 9），完成百分比 90%
- 不要自行改為「測試中」或「已完成」

### 步驟 7：記錄工作日誌
