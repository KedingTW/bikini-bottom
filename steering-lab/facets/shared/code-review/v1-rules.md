# Code Review 審查規則

## 審查原則

1. **只聚焦於本次 PR 的變更內容**（diff 中的程式碼）
2. 回饋僅列出**需要修正或有問題的部分**，符合規範的不需要額外列出
3. 如有必要，附上簡短程式碼範例協助理解
4. 不要對 diff 以外的既有程式碼提出意見
5. 第 2 輪起，不重複已被工程師合理拒絕的項目

## 嚴重程度分類

| 等級 | 標記 | 定義 | 不修正的後果 |
|------|------|------|-------------|
| 必修 | `[必修]` | 安全漏洞、明確 bug、會導致系統錯誤、資料遺失 | 不通過 |
| 建議 | `[建議]` | 可讀性改善、架構優化、最佳實踐、潛在風險 | 工程師給出理由即可接受 |
| 備註 | `[備註]` | 風格偏好、未來可改善的方向、知識分享 | 不影響通過與否 |

**通過條件：所有 [必修] 項目都已解決（修正或給出被接受的理由）。**

## 審查面向

### 1. 邏輯正確性與潛在錯誤

- 是否有邏輯錯誤、潛在 Bug 或未處理的邊界條件？
- 是否有不恰當的例外處理或錯誤忽略？
- 是否有未處理的 null/undefined/None 情況？
- 非同步邏輯是否正確處理（race condition、未 await 的 Promise 等）？

### 2. 安全性

- 是否有硬編碼的密鑰、token、密碼？
- 是否有 SQL injection、XSS、CSRF 風險？
- 使用者輸入是否有適當的驗證和清理？
- 是否有不安全的反序列化或檔案操作？

### 3. 可讀性與命名一致性

- 變數、函式、類別命名是否清晰且語意明確？
- 命名是否遵循該語言的慣例（見下方語言規範）？
- 是否有過於複雜、難以理解的邏輯需要重構或加註解？

### 4. 程式碼結構與重構潛力

- 是否有重複、冗餘的邏輯？
- 是否存在可抽取函式或模組化的機會？
- 函式是否過長（建議單一函式不超過 50 行邏輯）？
- 流程是否簡潔、易於維護？

### 5. 最佳實踐與風格一致性

- 是否符合常見設計原則（SRP、DRY、KISS）？
- 是否正確處理錯誤與資源釋放？
- 是否使用合適的資料結構？
- 是否與專案既有風格一致？

### 6. 分層設計（如適用）

根據專案的技術棧，檢查是否有跨層職責混淆：

**通用原則：**
- 路由/入口層：只處理請求轉發與參數驗證，不包含業務邏輯
- 業務邏輯層：專注業務邏輯，協調多個資料來源
- 資料存取層：專注資料存取，不包含業務邏輯
- 資料模型：只定義資料結構

**如果 repo 有 `.github/REVIEW_CONTEXT.md`，以該檔案定義的分層為準。**

## 語言命名規範

根據檔案副檔名判斷語言，適用對應的命名慣例：

### Python（`.py`）
- 變數/函式：snake_case
- 常數：UPPER_SNAKE_CASE
- 類別：PascalCase
- 私有成員：_leading_underscore

### JavaScript / TypeScript（`.js` / `.ts`）
- 變數/函式：camelCase
- 常數：UPPER_SNAKE_CASE
- 類別/元件：PascalCase
- 資料欄位/JSON key：camelCase 或 snake_case（依專案慣例）

### PHP（`.php`）
- 變數/函式：camelCase
- 常數：UPPER_SNAKE_CASE
- 類別：PascalCase
- 資料庫欄位：snake_case

### Vue（`.vue`）
- 元件名稱：PascalCase
- props/events：camelCase
- 其餘同 JavaScript

**如果 repo 有 `.github/REVIEW_CONTEXT.md` 定義了不同的命名規範，以該檔案為準。**

## Review Comment 格式

### 不通過時

```
## Code Review — 第 N 輪

### 審閱結果：❌ 需要修正

---

### [必修] 問題標題
**檔案**: `path/to/file.ext` L行號
**問題**: 具體描述問題所在
**建議**: 修改方向或範例

---

### [建議] 問題標題
**檔案**: `path/to/file.ext` L行號
**問題**: 具體描述
**建議**: 修改方向

---

### [備註] 問題標題
**說明**: 知識分享或未來改善方向

---

### 摘要
- 必修項目：N 項（必須修正才能通過）
- 建議項目：N 項（可選修正，給出理由即可跳過）
- 備註項目：N 項（僅供參考）

請修正 [必修] 項目後通知我再次審閱。
```

### 通過時

```
## Code Review — 第 N 輪

### 審閱結果：✅ 通過

本次變更符合程式碼品質標準，無需修正。

（如有 [備註] 項目，可選擇性列出供未來參考）
```

### 3 輪結束最終摘要

```
## Code Review — 最終摘要（3 輪已達上限）

### 已解決項目
- （列出已修正的項目）

### 未解決項目
- （列出仍未解決的項目及工程師的回覆理由）

### 結論
本次 review 已達 3 輪上限，請專案負責人決定後續處理方式。
```

## 避免過度審查的原則

為了避免無限輪迴和越改越壞的情況：

1. **[建議] 和 [備註] 不擋通過**：工程師回覆合理理由即接受
2. **不要求完美**：只要沒有 [必修] 問題就通過
3. **不擴大審查範圍**：每一輪只看未解決項目 + 新 commit 引入的問題
4. **接受合理的技術債**：如果工程師說「這個之後再處理」且不影響功能正確性，接受
5. **不重複提出**：工程師已回覆「不修改」+ 理由的項目，下一輪不再提
6. **修正引入新問題才延長**：如果新 commit 沒有引入新的 [必修] 問題，就通過
